Linux System Administration

A comprehensive collection of enterprise-grade Linux administration projects focused on user and group management, automated deployments, secure web hosting, centralized logging, access control, DNS, firewall configuration, and system monitoring. These projects showcase real-world implementations of scalable infrastructure, built for Dunder Mifflin’s multi-server environment, emphasizing automation, security, and reliability.

Project Overview

This section showcases a series of Linux system administration projects carried out for a fictional enterprise, Dunder-Mifflin. The goal was to build a robust, automated, and secure server infrastructure across multiple operating systems and network segments.

Network Topology

The network design includes a Rocky Linux router, DNS and web servers across both Rocky and Debian distributions, a file server using AlmaLinux, and a FreeBSD host. Machines are divided between a LAN (10.21.32.0/24) and a DMZ (100.64.N.0/24) segment.

Features

User Management Automation

To streamline operations at Dunder Mifflin, a centralized script was developed to automate user account creation and management across Linux (Machines A–F) and FreeBSD (Machine X) systems. Each user is assigned a standardized username, private group, valid shell, and home directory initialized from /etc/skel/. Passwords are synchronized across all systems, and home directories are restricted to individual users. Role-based secondary groups such as managers, sales, and accounting ensure appropriate access controls, while shared directories under /home/ on Machine E are secured with SetGID and proper ownership. Existing user files were migrated with correct permissions, sudo privileges were granted to administrators via group assignment, and umask settings were adjusted network-wide to maintain secure default permissions.

Dunder Mifflin Administrative Access Management

Administrative permissions were customized to align with Dunder Mifflin’s operational needs and security guidelines. Pam, Kelly, and Andy were configured for password-less SSH access to Machine C, improving efficiency without compromising safety. Dwight Schrute was granted unrestricted sudo access without group membership, adhering to the principle of least privilege. Michael Scott can schedule system shutdowns across DMZ machines with timed delays, including cancellation capabilities. Jim Halpert’s permissions allow ownership modification on Machine E, excluding the root account. These policies enforce granular control while preserving employee productivity and limiting potential misuse.

Multi-Site Apache Web Server Configuration

Apache was deployed on Machine C to host three company websites: dundermifflin.com, michaelscottpapercompany.com, and schrutefarms.com. Each site is reachable with or without the "www" prefix and stored under dedicated DocumentRoots. Access control policies enable Pam, Kelly, and Andy to manage the Dunder Mifflin site, while Dwight manages Schrute Farms. Machine D serves as a synchronized backup using rsync and cron. A CGI script on dundermifflin.com reports disk usage statistics and is secured via HTTP Basic Authentication and domain-based access controls, ensuring transparency while safeguarding sensitive tools.

Enhanced User Access and Password Policy Enforcement with PAM

Granular access control was achieved by configuring PAM rules across Linux systems. Only root, the system administrator, Michael, and Dwight can log in to all systems, with broader access allowed on Machine E. DMZ machines such as C, D, and F are restricted to specific departments based on their job roles. Password policies enforced via the pwquality module mandate a minimum of 10 characters, at least two uppercase letters, two digits, and one special character. Login times for most users on Machines C and D are restricted to weekdays between 9 AM and 5 PM, further tightening DMZ access.

Configuration Management with Ansible

Ansible was used to implement a robust configuration management system controlled from Machine E. Password-less SSH login was configured to all target nodes (A–F), and inventory groups were created for organization. The custom mkdmuserplay script generated a YAML playbook to replicate users, groups, and credentials across the network using data from passwd and shadow files. Umask configurations were standardized via a managed script, and additional playbooks ensured Apache was updated and configured to auto-start. These tools collectively established a reproducible, scalable, and administrator-friendly network configuration model.

Expanding Disk Storage with LVM and RAID

On Machine E, new drives were partitioned using Linux LVM to create flexible storage pools under a volume group named savg. Two logical volumes were allocated: /tmp (1GB, ext4, with noexec and nosuid) and /home (80% of remaining space, xfs, with nodev), each mounted via /etc/fstab to persist across reboots. RAID was configured beneath the volume group to ensure redundancy and resilience. These changes provide scalable and secure disk usage for both temporary and user data while adhering to best practices in filesystem security.

Automation of Network Configuration with DHCP and Hostname Updates

DHCP was centralized on Machine A to automate network settings and assign descriptive hostnames. Separate address pools were defined for LAN and DMZ ranges. Hostnames like dns0.dundermifflin.com, web0, and nfs were dynamically assigned to match each server's role. Machines B–F were configured to pull their settings from Machine A. DNS search domains and name server IPs were included in lease options. Once DHCP client settings were updated and systems rebooted, the entire infrastructure reflected accurate, role-based identity and networking information.

Accounting Directory Sharing with NFS and Network Time Sync

The accounting department’s need to contribute to the website was addressed by exporting a shared NFS directory from Machine E and mounting it on Machine C. Proper permissions ensured content could be read via the web server but not altered by unauthorized users. Web root and backup configurations were verified, and the automounter was set up for on-demand mounting to optimize resources. In parallel, time synchronization was established using Chrony on Machine A as a Stratum 2 server, with all other machines (including BSD and Debian nodes) syncing via appropriate NTP or systemd-timesyncd services.

DNS Server Hosting with BIND

A full-featured DNS infrastructure was implemented using BIND on Machine B (dns0), supporting both forward and reverse zones for LAN and DMZ. CNAME and A records were added for all hosts, with precise TTL, SOA, and zone transfer settings. Split DNS views allowed internal users access to complete records while external clients saw only filtered entries. DHCP settings were updated to use the new DNS infrastructure. Machine F served as a secondary name server with synchronized zones, ensuring redundancy and reliability. Reverse lookup zones completed the configuration for a professional, ISP-grade DNS setup.

Firewall Hardening with NFtables

NFtables replaced iptables across the infrastructure, applying default-deny policies and allowing only essential inbound services. Web servers (C and D) accepted HTTP while restricting outbound traffic, only permitting package updates, DNS, NFS, and SSH to known peers. DNS servers (B and F) accepted requests with carefully scoped zone transfer rights. The NFS server (E) allowed access only from authorized subnets. ICMP rules enabled diagnostics while banning Facebook by IP. All machines included SSH access rules and preserved local loopback functionality. Rules were made persistent with nftables.conf and validated through active network tests to ensure functionality and compliance.
